Topics of lectures:


Concurrency Testing: Challenges, Algorithms, and Tools

Madan Musuvathi, Microsoft Research.


The concurrency testing problem is to ensure that a concurrent program is correct for a given input. In this lecture, you will learn why this problem is hard, the practical implications of this for industrial-scale programs, and various research tools for addressing this problem. Two main approaches include a model-checking technique for providing absolute coverage guarantees and a randomized algorithm for providing probabilistic guarantees. Special attention will be paid to concurrency correctness criteria. We will provide tools that students can use to better understand the behavior of concurrent programs and essential correctness definitions.  A new version of the CHESS concurrency unit testing tool will be available, including monitors to check for: deadlocks, livelocks, data races, atomic method execution, linearizable method execution, deterministic behavior, temporal ordering properties.


Concurrent Algorithms and Transactional Memory

Mark Moir, Sun Microsystems


We will discuss concurrent data structures and algorithms for shared memory multicore systems, with an emphasis on the relationships between hardware support for synchronization (including existing and hypothetical future support) and various properties of implementations that can be achieved using it, including performance, simplicity, correctness and progress properties.


Analysis and Verification of Concurrent Programs

Ganesan Ramalingam, Microsoft Research


We will study (automated) analysis and verification of concurrent programs in this lecture. Specifically, we will focus on verifying safety properties of shared-memory imperative programs, with an emphasis on abstract-interpretation based techniques. We will use various concurrent data structure implementations as motivating examples. We will consider the verification of safety properties such as memory safety, data structure invariants, partial correctness, and linearizability. (Linearizability is a common and very useful correctness criterion for concurrent data-structures.) We will address issues introduced by fine-grained concurrency, a shared mutable heap, and the presence of an unbounded number of objects and threads. We will cover abstraction techniques and reduction techniques that help address these issues.

Low-level Concurrency - For Real

Peter Sewell, CAmbridge University


Concurrent algorithms are often described with respect to sequentially consistent shared memory.  However, real multiprocessors and concurrent programming languages incorporate a range of optimizations that have subtle programmer-visible effects, giving rise to relaxed memory models.  In some cases (such as Sparc TSO) these are captured in a well-defined way, making it possible (if challenging) to reason with confidence about the behavior of concurrent programs.  For others, however, it has been very unclear what a reasonable model is, despite extensive research over the last three decades.  In these lectures I will discuss the relaxed memory models of x86, Power and ARM multiprocessors, including their origins in hardware optimizations, the vendor architectural specifications, and how one can reason about concurrent algorithms above them.


Data-Parallel Programming of GPUs, multicore processors and FPGAs with Accelerator

Satnam Singh, Microsoft Research.


We present the Accelerator system which supports the compilation of high level data-parallel descriptions onto three very different kinds of execution environments: GPUs; vector instructions on multicore processors; and FPGA circuits. The ability to compile the same description onto three different kinds of target architecture allows us to program for future heterogeneous systems which will contain many different kinds of computing elements. These lectures present the design of a data-parallel programming system that captures programs at the right level of abstraction for compilation onto different targets. In particular, we describe how high level memory access operations in terms of whole array operations permit the efficient generation of code and circuits.


Deny-Guarantee Reasoning

Matthew Parkinson, Cambridge University


In this course, I will explain Deny-guarantee reasoning:  A new logic for program verification that integrates concepts from both separation logic and rely-guarantee reasoning. I will begin by presenting its foundations by giving an introduction to both separation logic and rely-guarantee reasoning.  Then, I will present deny-guarantee formally, and show that it captures the previous reasoning of both separation logic and rely-guarantee.  I will demonstrate how deny-guarantee enables verification of programs that use dynamic allocation and deallocation of threads, which was not possible with rely-guarantee.  Finally, I will show how to use deny-guarantee to enable abstraction of concurrent libraries.


Models, methods and tools for virtual prototyping embedded architectures

Jean Pierre Talpin, INRIA


This lecture will start with an introduction to the models of computation of synchronous data-flow languages and demonstrate the application of this formal modeling framework to virtual prototyping.  The activity of virtual prototyping aims at producing the executable specification of an embedded software architecture as soon as possible in its design flow, so as to check specified requirements against it, predict performances and validate architectural choices.  In this talk, we will show how to perform virtual prototyping using a synchronous data-flow language by showing how to interpret heterogeneous behavioral specifications (data-flows, automata, imperative programs) and structural descriptions (operating system services) into our model of computation. We will conclude by briefly illustrating some industrial applications of these concepts.


Software Transactional Memory

Adam Welc, Intel Research.


This lecture will provide a comprehensive overview of transactional memory: description of transactional language extensions and their semantics as well as discussion of important aspects of a software transactional memory (STM) system implementation. In particular, we will show how to extend popular programming languages, such as C/C++ or Java, with transactional constructs and how these constructs alleviate some of the problems related to programming with locks. We will also present the fundamentals of implementing STM algorithms both in the TM runtime and in the transactions-aware compiler.
